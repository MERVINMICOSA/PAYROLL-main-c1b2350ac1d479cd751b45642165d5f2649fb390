# Attendance Tab - System Analysis & Documentation Plan

## Overview
The attendance tab is the primary interface for managing employee daily attendance, teacher loads, and payroll inputs across multiple categories (EDA, SHS, College, Admin, Faculty, Guard, SA). It features dynamic tab-based navigation, period management, real-time data editing, and pagination.

## Architecture

### 1. Frontend (attendance.html)
- **Single Page Application** (~2500 lines of HTML/JS/CSS)
- 12 attendance tabs with different configurations
- Tailwind CSS for styling, Font Awesome for icons, Chart.js for graphs
- Vanilla JavaScript (no frontend framework)

### 2. Period Management System
- **PeriodManager class** (`js/period-manager.js`)
  - Handles period creation, validation, overlap prevention
  - Communicates with `/api/period/index.php`
  - Features: active period detection, date range validation, overlap checking
  - Local/session storage for persistence

### 3. Backend API Endpoints
- **Period Management**: `/api/period/index.php` (GET/POST/PUT/DELETE)
  - Types: all, active, specific, overlaps
  - Overlap prevention via SQL query
- **Attendance Data**: `/api/attendance/{tab}.php`
  - EDA: `/api/attendance/eda.php` - Employee Daily Attendance (admin staff)
  - SHS: `/api/attendance/shs-*.php` - SHS teacher loads & DTR
  - College: `/api/attendance/college-*.php` - College teacher loads & DTR
  - Admin/Faculty: `/api/attendance/admin-*.php` - Admin pay & master payroll
  - Guard/SA: `/api/attendance/guard.php`, `/api/attendance/sa.php`

### 4. Database
- PostgreSQL (via DATABASE_URL)
- Tables: `attendance_periods`, `attendance_eda`, `teacher_loads`
- `SecureDatabase` model class for safe queries

## Tab Configuration (TAB_CONFIG)

Each tab has:
- **title**: Display name
- **columns**: Table header columns
- **employeeFilter**: Function to filter employees by assignment
- **fields**: Database field names
- **editable**: Boolean array per field
- **hasDynamicDates**: Whether to generate date columns
- **dynamicColumns**: 'date' or 'checkbox' for dynamic rendering

### 10 Tabs:
1. **EDA** - Static table (lates, absences, overtime) for admin staff
2. **SHS Loading** - Static table with daily hours (Mon-Sun)
3. **SHS-DTR** - Dynamic date columns with hours per day
4. **College Loading** - Same as SHS Loading
5. **College-DTR** - Same as SHS-DTR
6. **Admin Pay** - Admin rate/hours for teachers
7. **Admin Master** - Full payroll with deductions
8. **Faculty SHS** - Teacher payroll summary
9. **Faculty College** - Teacher payroll summary
10. **Faculty Merge** - Combined summary (read-only)
11. **GUARD** - Checkbox-based daily attendance
12. **SA** - Checkbox-based daily attendance

## Core Functionality

### 1. Period Selection
- Set period via date inputs + "Add Period" button
- Creates period in DB (overlap validation)
- "Load Period" button shows modal with saved periods
- Filter by month/year
- Auto-load active period on page load

### 2. Data Loading
```
switchTab(tabId) → loadTabData()
  ├─ renderStaticTab()  → Fetches /api/attendance/{tab}.php
  │    ├─ Gets saved data for period
  │    ├─ Merges with employee list
  │    └─ Renders table with pagination
  │
  └─ renderDynamicDateTab() → Renders date columns per period day
       ├─ For DTR tabs: Shows hours per date
       └─ For Guard/SA: Shows checkboxes per date
```

### 3. Real-time Editing
- **Static cells**: `updateCellValue()` → POST all row fields
- **Daily hours**: `updateDailyHours()` → POST per-date hours
- **Checkboxes**: `updateCheckboxValue()` → POST present/absent
- Auto-saves on change with visual feedback
- Shows "Saving..." indicator
- Reloads tab data after save

### 4. Auto-calculations
- **Total Hours**: Sum of daily hours (DTR tabs)
- **Total Pay**: Days worked × rate (Guard/SA)
- **Gross Pay**: Basic + Overtime (payroll tabs)
- **Net Pay**: Gross - Deductions + Allowances (admin/master)

### 5. Pagination
- Client-side pagination (10/25/50/100 per page)
- Shows range (e.g., "1-10 of 150")
- First/Prev/Next/Last + page numbers

## Data Flow

### Loading:
1. `DOMContentLoaded` → Initialize PeriodManager
2. `loadEmployees()` → Fetch all employees from `/api/employees.php`
3. Try to load active period from PeriodManager (DB) → fallback to localStorage
4. `switchTab('eda')` → Load initial tab data

### Saving:
1. User edits cell → `onchange` triggers update function
2. Build payload with all row fields (preserve existing values)
3. POST to `/api/attendance/{tab}.php`
4. On success: show "Saved" toast, reload tab data
5. On error: show error toast

### Period Creation:
1. User clicks "Add Period"
2. `setPayrollPeriod()` → `periodManager.checkOverlaps()`
3. If no overlaps → `periodManager.createPeriod()`
4. POST to `/api/period/index.php`
5. Store in localStorage, update UI

## Key Features

1. **Overlap Prevention**: Periods cannot overlap (DB-level + app-level)
2. **Auto-save**: Changes saved immediately on input change
3. **Visual Feedback**: Color-coded cells, saving indicators, toasts
4. **Date Validation**: Periods validated on creation/load
5. **Multi-format Export**: JSON backup (period data) / Excel (coming soon)
6. **Keyboard Shortcuts**: Ctrl+S (save), Ctrl+E (export), Ctrl+H (help)
7. **Responsive Design**: Mobile-friendly layout
8. **Error Handling**: Network retry logic, error toasts

## Security

- Session-based authentication (PHPSESSID)
- CORS headers configured
- CSRF considerations (in separate modules)
- Input validation (date format, required fields)
- No credentials logged

## Known Limitations / Areas for Improvement

1. Excel export not implemented (placeholder)
2. "Copy previous week" not implemented
3. Some API endpoints return HTML instead of JSON on error
4. Pagination is client-side (could be server-side for large datasets)
5. No bulk undo/redo
6. No change history/audit log
7. Edit-all-cells-at-once could be more efficient (one request per cell)
