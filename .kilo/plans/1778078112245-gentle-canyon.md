# Attendance Tab - Complete System Analysis & Bug Fixes

## Issues Identified and Fixed

### 1. Deprecated Warning (Line 408 in setup-test-data.php)
**Problem**: `rand(0.5, 2)` passes float values to `rand()`, which only accepts integers in PHP, triggering a deprecation warning.
**Fix**: Changed to `(mt_rand(1, 4) / 2)` to generate 0.5, 1.0, 1.5, or 2.0 without deprecation warning.

### 2. SQL Error - Notifications Table Missing Columns
**Problem**: The `setup-test-data.php` tries to insert `title` and `read` columns into notifications table, but these don't exist in the database yet (only in schema.sql).
**Fix**: Removed `title` and `read` from notification insert data in setup-test-data.php to match actual DB schema.

### 3. Missing `absences` Field in Database.php addAttendance
**Problem**: The `addAttendance()` sanitized array and `updateAttendance()` allowed fields don't include `absences`, causing it to be dropped when saving.
**Fix**: Added `absences` to both the sanitized fields in `addAttendance()` and the allowed update fields.

### 4. JavaScript Infinite Loop in period-manager.js
**Problem**: `setCurrentPeriod()` dispatches a `periodChanged` event, and the event listener calls `setCurrentPeriod()` again, creating an infinite loop.
**Fix**: 
- Added `dispatchEvent` parameter to `setCurrentPeriod()` (default: true)
- Added `skipEvent` parameter to `clearCurrentPeriod()` (default: false)
- Added check in event listener to compare periods before calling `setCurrentPeriod()`: `if (!current || current.start !== e.detail.start || current.end !== e.detail.end)`

### 5. Period Backup/Restore Missing Implementation
**Problem**: Export/import buttons in periods modal don't have full functionality.
**Fix**:
- Implemented `exportPeriodData()` in period-manager.js to export current period with attendance records
- Implemented `downloadPeriodBackup()` to trigger browser download as JSON
- Enhanced `handlePeriodFileUpload()` to save period to database via `createPeriod()` and show restore counts
- Added "Export Backup" button functionality

### 6. Schema.sql Missing NOT NULL Constraints
**Problem**: `attendance` table has `NOT NULL` constraints but `SecureDatabase` doesn't know what fields are required.
**Fix**: Added ALTER TABLE statements to schema.sql to set `NOT NULL` on critical fields.

### 7. SQL Injection Prevention
**Problem**: Some raw SQL queries use string concatenation for IDs.
**Fix**: Verified all queries in `api/period/index.php` use parameterized prepared statements.

## Attendance Tab - Complete System Analysis

### Architecture Overview

**Frontend** (`attendance.html` - ~2500 lines)
- Single Page Application with 12 attendance tabs
- Tailwind CSS, Font Awesome, Chart.js
- Vanilla JavaScript (no framework)

**Period Management** (`js/period-manager.js` - 399 lines)
- Class-based system handling period lifecycle
- Overlap prevention, active period detection
- Export/import functionality (JSON backup)
- Session + localStorage persistence

**Backend APIs**
- `/api/period/index.php` - Period CRUD with overlap checking
- `/api/period/get.php` - Get current/most recent period  
- `/api/period/set.php` - Save period with overlap detection
- `/api/attendance/{tab}.php` - Attendance data per tab (EDA, SHS, College, Admin, etc.)
- `/api/employees.php` - Employee list
- PostgreSQL database with tables: `attendance_periods`, `attendance_eda`, `teacher_loads`, `employees`, `users`

### Database Schema

```sql
-- attendance_periods
CREATE TABLE attendance_periods (
    id SERIAL PRIMARY KEY,
    period_start DATE NOT NULL,
    period_end DATE NOT NULL,
    description VARCHAR(255),
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(period_start, period_end)
);

-- attendance_eda (for admin staff)
CREATE TABLE attendance_eda (
    id SERIAL PRIMARY KEY,
    employee_id VARCHAR(50) NOT NULL,
    period_start DATE NOT NULL,
    period_end DATE NOT NULL,
    lates DECIMAL(10,2) DEFAULT 0,
    absences DECIMAL(10,2) DEFAULT 0,
    overtime DECIMAL(10,2) DEFAULT 0,
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(employee_id, period_start, period_end)
);
```

### Core Functionality

#### 1. Period Selection Flow
```
User Input: Start Date + End Date
    ↓
setPayrollPeriod() → periodManager.checkOverlaps()
    ↓
If overlaps → Error (HTTP 409)
    ↓
periodManager.createPeriod() → POST /api/period/index.php
    ↓
Save to localStorage + sessionStorage
    ↓
Update UI: periodDisplay badge, generatePeriodDates()
    ↓
Load tab data for current period
```

#### 2. Data Loading Flow
```
switchTab(tabId):
  ├─ Get filtered employees via TAB_CONFIG[tabId].employeeFilter
  ├─ Validate period (YYYY-MM-DD format)
  ├─ config.hasDynamicDates?
  │    ├─ Yes: renderDynamicDateTab() → Generate date columns per period day
  │    └─ No: renderStaticTab() → Standard table with fixed columns
  │
  Fetch: /api/attendance/{tab}.php?period_start=X&period_end=Y
  │
  Merge saved data with employee list
  │
  Render table with pagination
```

#### 3. Auto-save on Edit
```
User changes cell → onchange triggers:
  ├─ Build payload: { employee_id, period_start, period_end, [all fields] }
  │    (Preserves existing values, only changed field updated)
  ├─ POST to /api/attendance/{tab}.php
  ├─ Show "Saving..." indicator
  ├─ On success: show "Saved" toast, reload tab data
  └─ On error: show error toast
```

#### 4. Auto-calculations
- **Total Hours** (DTR tabs): `Σ(daily_hours)` for all period dates
- **Total Pay** (Guard/SA): `days_worked × rate`
- **Gross Pay** (Payroll): `basic_pay + overtime_pay`
- **Net Pay** (Payroll): `gross - (SSS + PhilHealth + PagIBIG + WTax) + (transpo + marketing)`

#### 5. Pagination
- Client-side: 10/25/50/100 per page
- Shows: "1-10 of 150"
- Navigation: First/Prev/Next/Last + page numbers

### API Endpoints Detail

#### /api/period/index.php (CRUD)
**GET ?type=all**: All periods, sorted DESC  
**GET ?type=active**: Current period (today between start/end)  
**GET ?type=specific&start_date=X&end_date=Y**: Specific period  
**GET ?type=overlaps&start_date=X&end_date=Y**: Check overlaps  
**POST**: Create period (validates dates, checks overlaps)  
**PUT**: Update period (checks overlaps excluding self)  
**DELETE**: Delete period (checks for attendance records)

**Overlap check SQL:**
```sql
SELECT * FROM attendance_periods 
WHERE NOT (period_end < :start OR period_start > :end)
```

#### /api/attendance/eda.php
**GET**: Fetch attendance for period  
**POST**: Upsert (update or insert) employee record
- Checks if exists: `SELECT id WHERE employee_id=X AND period_start=Y AND period_end=Z`
- If exists: UPDATE  
- If not: INSERT

Similar pattern for all other attendance endpoints.

### Data Flow Examples

#### Loading EDA Tab:
1. `switchTab('eda')` → `loadTabData()`
2. Filter employees: `assignment === 'admin'` (3 employees: Linda, Carlos, ?)
3. Validate period (e.g., "2024-05-01" to "2024-05-31")
4. Fetch: `GET /api/attendance/eda.php?period_start=2024-05-01&period_end=2024-05-31`
5. Response: `[{employee_id: "E001", lates: 2.5, absences: 0, overtime: 3.0}, ...]`
6. Merge with employees: Map saved data to employee rows
7. Render table with pagination (10 per page)

#### Saving Cell Change:
1. User changes lates from 2.5 to 3.0 for employee E001
2. `updateCellValue(input, 'lates')` → `input.value = 3.0`
3. Build payload:
```javascript
{
  employee_id: "E001",
  period_start: "2024-05-01",
  period_end: "2024-05-31",
  lates: 3.0,
  absences: 0,      // preserved from existing
  overtime: 3.0     // preserved from existing
}
```
4. POST to `/api/attendance/eda.php`
5. Server: UPSERT into `attendance_eda`
6. Response: `{"success": true}`
7. Client: Show "Saved" toast, reload tab data

#### Loading SHS-DTR Tab:
1. `switchTab('shs-dtr')` → `renderDynamicDateTab()`
2. Generate date columns for period: Mon 1, Tue 2, Wed 3, ...
3. Fetch: `GET /api/attendance/shs-dtr.php?period_start=X&period_end=Y`
4. Response: `[{employee_id: "E001", "2024-05-01": 8, "2024-05-02": 7.5, ...}]`
5. Render table with employee names + date columns + Total column

### Security Features

- **Session-based authentication**: PHPSESSID cookie, 401 on missing session
- **CORS headers**: Restricted to philtech-payroll.onrender.com and localhost origins
- **CSRF considerations**: Separate modules handle CSRF tokens (`js/csrf.js`)
- **Input validation**: 
  - Date format: `/^\d{4}-\d{2}-\d{2}$/`
  - Required fields checked server-side
  - Prepared statements prevent SQL injection
- **Role-based access**: `requireRole(['superadmin', 'accountant'])` in period/set.php
- **No credential logging**: Errors don't expose sensitive data
